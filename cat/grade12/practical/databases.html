Databases (Access) Grade 12

Grade 12 extends Access to advanced queries (calculated fields, totals, wildcards, IS Null), input masks, attachment fields for photos, lookup fields from a typed list vs. another table, grouped reports with calculations, and designing complete database solutions for specific scenarios.

Advanced Queries

Calculated Fields in Queries

You can add a new column in a query that calculates a result based on other fields:

Calculated field syntax in query design
In the Field row of a blank column, type:
FieldName: [Field1] * [Field2]

Examples:
TotalPrice: [UnitPrice] * [Quantity]
Discount: [Price] * 0.1
FullName: [FirstName] & " " & [LastName]
Age: Year(Now()) - Year([DateOfBirth])

Totals / Group By in Queries

The Totals row allows you to perform aggregate calculations grouped by a field:

  • Query Design → Design tab → Totals (Σ) button
  • A Total row appears under each field
  • Set it to Group By for the grouping field
  • Set it to a function (Sum, Avg, Count, Max, Min) for calculated fields
Function in Total rowEffect
Group ByGroups records by unique values in this field
SumAdds all values in the field for each group
AvgAverage value for each group
CountNumber of records in each group
MaxLargest value in each group
MinSmallest value in each group
WhereUsed to filter records before grouping (criteria applies to ungrouped data)

Wildcard Characters

WildcardMatchesExampleFinds
*Any number of charactersLike "S*"Smith, Sarah, SA
?Any single characterLike "B?g"Big, Bag, Bug
#Any single digitLike "200#"2001, 2002, 2009

IS Null / IS Not Null

  • Is Null — finds records where the field is empty (no data entered)
  • Is Not Null — finds records where the field has any value

AND / OR / NOT in Queries

LogicHow to enter in query grid
AND Put criteria for BOTH fields in the same Criteria row
OR Put one criterion in Criteria row, the other in the Or row below
NOT Use Not "value" or <>"value" in the criteria

Advanced Reports — Grouped Reports

Step-by-step: Create a grouped report with subtotals
  1. Click the Create→Report Wizard button.
  2. Choose the table or query, then move the fields you want into the report (click each field, then click the > button). Click Next.
  3. Now choose what to group by : click the field (e.g. "Grade"), then click the > button to add it as a grouping level. Click Next.
  4. Choose how to sort the records inside each group (e.g. by Surname). Then click the Summary Options… button.
  5. Tick the boxes for the calculations you want per group — Sum, Avg, Count, Min or Max . Click OK, then Next.
  6. Choose a layout (e.g. "Stepped") and orientation, click Next, give the report a name, and click Finish.

Changing a Report's Source

To change which table or query a report is based on:

  1. Open report in Design View
  2. View the Property Sheet: Design → Property Sheet (F4)
  3. On the Data tab, find Record Source → click dropdown → choose different table or query

Data Validation in Tables

Validation ensures only correct data can be entered. Think of it as a bouncer at the door of each field — the Validation Rule decides what's allowed in, and the Validation Text is the message shown to anyone turned away:

PropertyExampleEffect
Validation Rule>=0 And <=100Marks must be between 0 and 100
Validation Text"Mark must be 0–100"Error message shown when rule is broken
Required = Yes(tick Yes)Field cannot be left empty
Input Mask000-0000000Forces phone number format
EXAM TIP — Rule and Text must match

Always write the Validation Text so it actually describes the Validation Rule. If the rule is >=0 And <=100 but the text says "must be a positive number", a mark of -5 and a mark of 500 would both fail the rule, but the message wouldn't make sense for the second case. Read the rule back to yourself in plain English, then write the text to match it.

Input Masks — Controlling Exactly What Can Be Typed

An input mask is stricter than a validation rule: instead of checking data after it's typed, it controls what you're allowed to type as you type it — similar to how a cellphone number field on a website only lets you type digits, automatically adding the dashes for you. Each position in the mask is built from special characters that mean "a digit must go here", "a letter must go here", and so on.

CharacterMeaning
0 A digit — required (0–9 only, no + or −)
9A digit or blank — optional
L A letter A–Z — required
?A letter A–Z — optional
AA letter or digit — required
aA letter or digit — optional
&Any character — required
>Forces every letter that follows to CAPITALS
<Forces every letter that follows to lowercase
\ Shows the next character literally (e.g. \A always displays the letter "A", it's not a placeholder)
"text"Anything inside quotes is displayed exactly as typed, every time (useful for a fixed prefix)
Worked example: a code that must read "NED26" + 2 capital letters + 3 digits
  1. The fixed part "NED26" never changes, so it's typed literally in quotes: "NED26"
  2. Two required capital letters next: each letter is L , and > beforehand forces capitals, so that's >LL
  3. Three required digits at the end: 000
  4. Joined together, the full input mask is: "NED26">LL000
BUILD IT PIECE BY PIECE

Don't try to write the whole mask in one go. Break the required pattern into chunks exactly as the question lists them, write the mask character for each chunk separately, then join the chunks in order. This is the same "piece by piece" approach used for nested IFs and combined text formulas above.

Lookup Fields — Two Different Sources

A lookup field shows a drop-down list when entering data, but that list can come from two different places — and exams test both, so it's worth knowing which is which.

TypeWhere the list comes fromWhen to use it
Value List You type the exact list of options yourself (e.g. Yes;No;Expires Soon ) when setting up the lookupFixed options that will basically never change, like Yes/No fields
Table/Query lookupThe list pulls live from another table or query (e.g. a Products table)Options that already exist as records elsewhere and might be added to later, like product names or class lists
Step-by-step: Create a typed Value List lookup
  1. Open the table in Design View, click the field, and open the Lookup tab of the Field Properties.
  2. Set Display Control to Combo Box.
  3. Set Row Source Type to Value List.
  4. In Row Source, type each option separated by a semicolon, e.g. Yes;No;Expires Soon.
  5. If users must only pick from this list (no typing their own value), set Limit To List to Yes.

Storing Photos — Attachment Fields

A normal text field can't hold a photo. To let a table store an actual image file against each record (e.g. a passport photo per learner), you need a field with the Attachment data type — Access then keeps the picture inside the database itself.

Step-by-step: Add a field that stores a photo
  1. In Design View, click into a new empty row below the field you want it after, and type the field name (e.g. Photo).
  2. Click into the Data Type column and choose Attachment from the dropdown.
  3. If a description is required, type it in the Description column — this text shows in the status bar when the field is selected in Datasheet View, helping anyone entering data understand what's expected.
  4. In Datasheet View, double-click the small paperclip icon in that field to browse to and attach the actual image file.

Import and Export

  • Import Excel: External Data → Excel → choose file → map columns to fields
  • Import CSV: External Data → Text File → follow wizard
  • Export to Excel: right-click table/query → Export → Excel
  • Export to CSV: right-click → Export → Text File
  • Linking: External Data → Excel → Link to data source (live link — changes in Excel reflect in Access)

Integration with Other Applications

  • Use an Access query as a data source for Excel (Data → Get Data → From Database)
  • Use an Access table as a mail merge data source in Word
  • Export a report to PDF: right-click report → Export → PDF or XPS
EXAM TIP — Designing a Database for a Scenario

You may be asked to design a database from scratch for a given scenario. Steps: (1) Identify the entities (what needs to be stored — e.g. Students, Courses). (2) List the fields for each table. (3) Identify the primary key. (4) Identify calculated fields. (5) Plan queries and reports needed. Always think: "What data do I need to capture? What reports will be generated?"
