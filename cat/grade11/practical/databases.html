Databases (Access) Grade 11

Microsoft Access is the database application used in CAT. Databases store structured data in tables and allow powerful filtering, querying and reporting. Grade 11 introduces tables, forms, queries and basic reports.

What is a Database?

A database is an organised collection of related data stored in a structured way so it can be efficiently accessed, managed and updated. A DBMS (Database Management System) is the software used to manage databases. Microsoft Access is a desktop DBMS.

Key Terminology

Table
The fundamental storage object — contains data in rows and columns (like a spreadsheet, but structured)
Record
A complete set of data about one item/person — one row in a table
Field
A single category of information — one column in a table (e.g. FirstName, DateOfBirth, Price)
Field name
The label/heading of a column (e.g. "StudentID", "Mark")
Primary key
A field (or combination) that uniquely identifies each record — no two records can have the same value (e.g. ID number, StudentCode)
Form
A user-friendly interface for entering, viewing and editing records (one record at a time)
Query
A question asked of the database — returns specific records that meet set criteria
Report
A formatted printable summary of data, often grouped and with calculations

Input Masks — Controlling How Data Is Typed

Think about a cellphone number field. Without any guidance, one person might type 0821234567, another 082 123 4567, and another (082) 123-4567 — all the same number, three different formats. An input mask forces everyone to type data the same way by deciding, character by character, what's allowed in each position.

Mask CharacterMeaning
0A digit (0–9) is required
9A digit is optional
LA letter is required
?A letter is optional
>Converts the following characters to UPPERCASE
<Converts the following characters to lowercase
\The next character is shown as-is (a literal symbol, not a rule)
Step-by-step: Add an input mask to a field
  1. Open the table in Design View and click the field that needs the mask.
  2. In the Field Properties panel at the bottom, click into the Input Mask row.
  3. Type the mask using the characters above. For example, 000-000-0000 forces exactly 10 digits with dashes already in place.
  4. Save the table (Ctrl+S) and switch to Datasheet View to test it — try typing in the field and watch how it guides you.
INPUT MASK vs VALIDATION RULE

An input mask controls the shape of what's typed (how many digits, where the dashes go) as you type it. A validation rule checks the value after it's typed (e.g. is the mark between 0 and 100). Exams sometimes test both on the same field — read carefully which one is being asked for.

Creating Tables

Step-by-step: Create a new table in Design View
  1. At the top of the screen, click the Create→Table Design button. A blank grid opens.
  2. In the first row under Field Name, type the name of your first field (for example LearnerID). Field names should have no spaces.
  3. Press Tab to move to the Data Type column. Click the small arrow and choose the correct type (e.g. Short Text, Number, Date/Time).
  4. Repeat for each field, one per row, until all your fields are listed.
  5. Click the row that should be the primary key (e.g. LearnerID), then click the Primary Key button (a small key icon) on the Design tab. A little key appears next to that row.
  6. To set rules for a field, click it, then change its Field Properties in the panel at the bottom of the screen (field size, default value, required, validation rule, etc.).
  7. Press Ctrl + S to save. Type a name for the table (e.g. tblLearners) and click OK.
  8. Switch to Datasheet View (the grid icon, top-left) to start typing in records.

Datasheet View

  • Add/edit/delete records directly in the table grid
  • Sort: click column header → Home → Sort Ascending/Descending
  • Filter: Home → Filter button or right-click a value

Creating Forms

Forms provide a friendly data entry interface.

  • AutoForm: select table → Create → Form (creates a basic form automatically)
  • Form Wizard: Create → Form Wizard → choose fields and layout
  • Design View: full control; add/remove fields, labels, buttons, images

Form Controls

Text Box
Displays/enters data from a field
Label
Static descriptive text (field name)
Button
Runs a macro or action (e.g. Save Record, Go to Next Record)
Image / Picture
Displays a logo or image on the form

Creating Queries

Queries retrieve specific records from a table based on criteria.

Step-by-step: Build a query to find all Grade 11 learners
  1. Click the Create→Query Design button.
  2. A "Add Tables" panel appears on the right. Double-click the table you want (e.g. tblLearners), then close the panel.
  3. The table appears with a list of its fields. Double-click each field you want to see in the result — they drop into the grid at the bottom.
  4. To filter, find the Criteria row in the grid, under the field you want to filter. Under "Grade", type 11.
  5. Click the big red Run button (exclamation mark !) on the Design tab. Access shows only the records that match.
  6. Press Ctrl + S to save, give the query a name (e.g. qryGrade11) and click OK.
REMEMBER: AND vs OR

To require TWO conditions at once (AND), type both in the same Criteria row. To allow EITHER condition (OR), type the second one in the "or" row just below.

AND vs OR in Queries

  • AND: place criteria in the same Criteria row — both conditions must be true
  • OR: place criteria in different Criteria rows — either condition can be true

Creating Reports

Reports present data in a formatted, printable way.

  • AutoReport: select table/query → Create → Report
  • Report Wizard: Create → Report Wizard → choose fields, grouping, sorting and layout

Report Sections (Design View)

Appears once at the very top of the report (title, logo, date)
Page Header
Appears at the top of every page (column headings)
Detail

Appears at the bottom of every page (page number)

Designing a Database for a Specific Purpose

Before opening Access at all, a good database starts on paper: what information needs to be stored, and how is it split into separate, sensible tables? A common mistake is cramming everything into one table — e.g. storing a learner's name AND every subject they take in the same row, which forces you to repeat the learner's name once per subject. Splitting into a Learners table and a Subjects table, linked by a shared LearnerID, avoids that repetition.

One table, one topic
Each table should describe one type of "thing" — Learners, Subjects, Results — not a mix
Avoid repeating data
If you find yourself typing the same name or address over and over, that data belongs in its own table, linked by an ID
Link tables with a common field
Usually the primary key of one table appears in the other as a "foreign key"

Troubleshooting a Database

ProblemLikely cause / fix
Can't type a record into a tableCheck whether a required field was left blank, or whether a validation rule is rejecting the value
Query returns no resultsCheck criteria spelling, check AND/OR rows are placed correctly, check for extra spaces in typed criteria
Report shows "#Error" in a calculationCheck the field name inside the calculation matches the actual field name exactly, including spelling
Can't delete a recordThe record may be linked to data in another table (referential integrity) — delete the related records first
EXAM TIP — Database Terminology

Know the difference: Table (stores data), Form (data entry interface), Query (filters/retrieves data), Report (formatted printed output). A query returns records from tables; a report presents those records nicely. Use Like "S*" for starts with, Like "*son" for ends with, Like "*an*" for contains "an".
